# 02 技术选型

本文定义 USB 安全管理系统的技术栈、部署方式和测试技术方案。选型内容面向研发、测试和交付实施，重点说明选型结论、适用范围、采用依据、工程约束、替换影响和自动化测试要求。

---

## 2.1 装置端服务语言

**选择：Rust**

| 维度 | 说明 |
|---|---|
| 适用范围 | 装置端常驻服务、USB Gadget/configfs 管理、HID 事件处理、Linux 挂载层生命周期管理、NBD 虚拟块设备服务、策略执行、私有协议服务、SQLite 数据访问。 |
| 选型依据 | 装置端属于长期运行的嵌入式系统级服务，核心要求为内存安全、资源生命周期可控、低延迟事件处理和跨 ARM64 构建稳定性。Rust 可在无 GC 的前提下提供编译期内存安全，适合处理 USB、块设备、mount、ioctl、Unix socket 等系统资源。 |
| 替代方案评估 | C/C++ 可满足系统级能力，但内存安全、并发安全和异常路径资源回收需要较高人工保障；托管运行时语言在 HID 事件转发和系统资源生命周期控制上存在额外运行时不确定性。 |
| Rust 版本 | **stable 1.75+**，使用 rustup 管理；推荐锁定到项目 `rust-toolchain.toml` 确保构建可复现 |
| 关键依赖库 | `tokio`（异步运行时）、`prost`（Protobuf 序列化）、`rustls` + `tokio-rustls`（纯 Rust TLS，无 OpenSSL 依赖）、`rusqlite`（SQLite，启用 bundled-sqlite3 feature 静态链接）、`nix`（mount、ioctl、系统调用封装）、`evdev`（读取 /dev/input/eventX）、USB 描述符解析相关 crate/接口（读取设备描述符、配置描述符、接口类、端点能力）、`tracing` + `tracing-subscriber`（结构化日志）、`serde` + `serde_json`（序列化）、`anyhow` + `thiserror`（错误处理）、`sd-notify`（systemd watchdog）|
| 构建要求 | `rustup target add aarch64-unknown-linux-gnu`；构建环境安装 `gcc-aarch64-linux-gnu`；`Cargo.toml` 固定目标 linker；产物进入装置端安装包，不在现场执行源码编译。 |
| 替换成本 | 高（需重写装置侧全部服务逻辑） |
| 安全准入要求 | USB 设备识别必须基于 udev、USB 描述符、接口类、接口子类、协议号和端点能力，不按设备名称、厂商字符串、产品字符串直接放行；普通 U 盘、键盘、鼠标按 PRD 定义分别进入对应准入和映射流程，不支持或无法识别的设备默认拒绝映射。 |
| 自动化测试要求 | 业务逻辑、状态机、策略判断、协议编解码必须可脱离真实 USB 设备运行单元测试；系统资源访问通过 trait/interface 抽象，支持 mock 注入；异步逻辑使用 `tokio::test` 覆盖超时、取消和资源释放路径；设备识别测试必须覆盖伪装设备、不支持设备、无法识别设备，以及白名单不放行键盘、鼠标或其它非普通 U 盘设备。 |

## 2.2 装置端数据存储

**选择：SQLite**

| 维度 | 说明 |
|---|---|
| 为什么选 | 嵌入式数据库，零独立进程，单文件存储，掉电后自动持久化。白名单、策略、日志的数据量不会大到需要独立数据库服务。Rust 有成熟的 `rusqlite` 驱动，启用 `bundled-sqlite3` feature 可静态链接 sqlite3，无外部 .so 依赖，交叉编译干净。 |
| 不选 PostgreSQL/MySQL | 需要独立数据库进程，占用额外系统资源，嵌入式离线设备的安装、备份和故障恢复复杂度较高。 |
| 当前阶段必要性 | 必要 |
| 替换成本 | 中（SQL 语句基本兼容，替换驱动即可） |
| 自动化测试要求 | 数据访问层必须支持 SQLite 内存模式（`:memory:`）和测试事务隔离；日志滚动、用户锁定、策略保存等规则需提供可重复执行的数据夹具。 |

## 2.3 装置端文件访问控制实现

**选择：Linux 挂载层 + Rust NBD 虚拟 exFAT 服务器**

| 维度 | 说明 |
|---|---|
| 方案组合 | **一层用户态块设备结构**：Linux 挂载层将真实 U 盘挂载到 `/mnt/usb_raw`；Rust NBD server 读取 `/mnt/usb_raw` 的目录结构和文件内容，应用病毒标记、可执行程序访问控制、文件类型黑名单、介质自动读取控制和只读权限，并动态生成虚拟 exFAT 扇区暴露给 `f_mass_storage` gadget。 |
| 为什么不用完整镜像 + loop | 需要在 RK3568 上预先复制完整 U 盘内容，32GB 存储空间不足以应对大容量 U 盘（128GB+）。 |
| 为什么不用 ublk | `ublk` 用户态块设备驱动更简洁，但需要 kernel 6.0+；RK3568 使用 kernel 4.19.232，不支持。 |
| 为什么选 NBD | kernel 4.19 原生支持 NBD 模块；Rust NBD server 可按 LBA 扇区请求动态返回虚拟 exFAT 内容；`f_mass_storage` 可直接绑定 `/dev/nbdX` 为 LUN；无需在 RK3568 上生成完整镜像。 |
| 完整数据链路 | 真实 U 盘块设备（`/dev/sdX`）→ Linux 挂载层挂载到 `/mnt/usb_raw`（目标支持 FAT32、exFAT、NTFS、ext2、ext3、ext4）→ Rust NBD server（策略判断 + 虚拟 exFAT 扇区生成）→ `/dev/nbdX` → f_mass_storage LUN → 受控主机（免驱，Windows/Linux 均支持）|
| 真实 U 盘文件系统支持边界 | 真实 U 盘文件系统目标支持 FAT32、exFAT、NTFS、ext2、ext3、ext4，实际支持范围以装置端系统镜像启用的内核驱动和挂载工具为准。文件系统无法识别、挂载失败、文件系统损坏或内容无法转换为受控 exFAT 视图时，该次 U 盘禁止映射并记录 USB 审计日志。 |
| Linux 挂载层职责 | 由 Linux 内核驱动或必要的用户态挂载工具识别并挂载真实 U 盘文件系统，向装置端服务提供稳定的目录树和文件内容访问入口。病毒扫描和病毒文件标记基于 `/mnt/usb_raw` 对应的受控文件树执行。 |
| NBD server 职责 | 读取 `/mnt/usb_raw` 的目录结构和文件内容，构建内存中的受控文件树；对受控主机发出的 exFAT 扇区 READ/WRITE 请求执行策略判断；未命中策略时返回真实内容，读写权限允许时将新增、修改、删除、重命名等变更同步写回真实 U 盘；命中阻断策略时返回 I/O error 或保护目录项，使受控主机弹出输入/输出错误类提示。 |
| 输出视图语义 | 受控主机侧看到的是装置端生成的虚拟 exFAT 块设备视图，不直接访问真实 U 盘文件系统；虚拟 exFAT 视图承载普通文件和目录访问，不承诺保留原始文件系统类型、NTFS ACL、Linux 权限位、属主属组、符号链接、硬链接、特殊文件、扩展属性等高级文件系统语义。 |
| 大文件处理 | 虚拟输出格式采用 exFAT，支持超过 4GiB 的单文件在受控视图中表达和读取；文件访问仍受病毒标记、可执行程序控制、文件类型黑名单、介质自动读取控制和只读权限约束。 |
| 病毒文件处理 | 病毒命中文件在虚拟 exFAT 目录中可见，文件名按 PRD 添加 `[病毒禁止访问]` 前缀，目录项文件大小为 0，不修改真实 U 盘上的原始文件名和内容；受控主机对病毒文件的读、写、修改、删除、重命名、执行、复制等操作均被禁止，目录项修改请求不得落到真实 U 盘。 |
| 阻断效果 | 被阻断文件在目录中可见；病毒文件按 size=0 与全操作禁止处理；可执行程序、文件类型黑名单和介质自动读取命中文件在用户打开或读取时由 NBD server 对对应数据扇区返回 I/O error，受控主机显示读取文件出错或输入/输出错误。整盘只读通过 `f_mass_storage` write-protect flag 实现，不依赖真实 U 盘文件系统权限位。 |
| 存储占用 | 虚拟 exFAT 目录树元数据保存在内存，文件数据按需从 `/mnt/usb_raw` 流式读取，RK3568 存储上不产生完整数据副本。 |
| 不选内核模块 | 内核模块开发复杂，RK3568 kernel 4.19 适配成本高，升级内核风险大。 |
| 关键 crate / 模块 | 内核 `mount` / `umount`、内核 `nbd` 模块（`/dev/nbdX`）、exFAT 目录项与簇分配生成模块、`nix`（系统调用封装） |
| 自动化测试要求 | 策略判断、文件树构建、exFAT 目录项生成、簇分配、超过 4GiB 单文件的正常映射与读取、策略命中 I/O error 判定需拆分为可单测模块；NBD 扇区读写、受控主机错误提示和 f_mass_storage 绑定需进入硬件端到端自动化测试设计。 |

## 2.4 病毒扫描引擎

**选择：ClamAV（clamd 守护进程 + clamdscan 客户端）**

| 维度 | 说明 |
|---|---|
| 为什么选 | PRD 明确指定 ClamAV。ClamAV 开源、支持 ARM 平台，可在 Ubuntu 22.04 / RK3568 环境运行。clamd 守护进程扫描性能优于每次启动 clamscan，适合作为装置端常驻扫描服务。 |
| 架构方式 | 装置侧 Rust 服务通过 Unix socket 与 clamd 通信，提交扫描任务，获取结果。 |
| 工程约束 | 需要在 RK3568 目标设备上验证 ClamAV 的扫描性能（大容量 U 盘扫描耗时）、内存占用和病毒库更新机制。PRD 第 7.8 节已说明此风险。 |
| 当前阶段必要性 | 必要 |
| 自动化测试要求 | 使用 EICAR 测试文件覆盖扫描命中链路；扫描接口需支持 Mock Scanner 注入，覆盖正常、命中、扫描失败、clamd 不可用、扫描中断等分支。 |

## 2.5 管理端应用形态

**选择：Electron 32.x + Vue 3.4.x**

**版本锁定：**

| 组件 | 版本 | 说明 |
|---|---|---|
| Electron | **32.x**（内置 Node.js 20.x，Chromium 128）| 最低要求 Windows 10 1809+，兼容 Win10/11 专业版和家庭版；不依赖组策略、Hyper-V 等专业版特性 |
| Vue | **3.4.x** | Composition API 稳定版，`<script setup>` 语法，与 Element Plus 兼容 |
| Vite | **5.x** | Electron + Vue 构建工具，替代 webpack |
| TypeScript | **5.x** | Vue 3 生态默认语言 |

| 维度 | 说明 |
|---|---|
| 为什么选 | PRD 要求 Windows 桌面客户端，需要本地文件对话框（选择升级包、策略文件、授权文件、日志导出路径）和与原型图一致的桌面 GUI。Electron 提供桌面能力，Vue 3 提供组件化页面开发能力，能够复用 Web 前端生态完成表格、表单、弹窗、路由和自动化测试。 |
| 不选 WPF/WinForms | UI 开发效率、组件生态和跨端自动化测试工具链不如 Electron + Vue 组合完整；与原型图 HTML 的视觉还原和组件复用成本较高。 |
| 不选纯浏览器 Web | 需要本地文件系统操作，纯浏览器受沙箱限制，交互体验差；PRD 明确要求桌面客户端程序。 |
| Windows 版本兼容性 | Electron 32 最低 Windows 10 1809（Build 17763），Win10/11 专业版和家庭版均无特殊限制 |
| Windows 中文环境兼容性 | 管理端需兼容 Windows 简体中文、繁体中文系统区域和中文用户名路径。页面文本、协议字符串、JSON、Protobuf string、SQLite 文本字段、导出文件名统一按 UTF-8/Unicode 处理，不使用 GBK、Big5 或 ANSI API 做手工转码。 |
| 文件路径编码要求 | Electron 主进程处理升级包、病毒库包、策略文件、授权文件、日志导出目录时必须使用 Unicode 路径能力，支持简体中文、繁体中文、空格和常见符号路径；禁止通过 shell 拼接未转义路径。 |
| 当前阶段必要性 | 必要 |
| 替换成本 | 高（UI 层需重写） |
| 自动化测试要求 | 页面组件使用 Vitest + Vue Test Utils 覆盖；Electron E2E 使用 Playwright 覆盖登录、授权、权限、策略配置、日志查询、连接断开提示、文件选择流程，以及简体/繁体中文路径下的导入、导出和安装路径兼容性。 |

## 2.7 管理端状态管理

**选择：Pinia**

| 维度 | 说明 |
|---|---|
| 为什么选 | Vue 3 官方推荐状态管理方案。管理端需要在进程内维护装置连接状态、当前登录用户、角色、session token、授权状态和当前页面状态。Pinia API 简洁、类型支持完整，适合按页面和业务域拆分 store。 |
| 不选 Vuex | 已被 Pinia 取代，API 冗余。 |
| 当前阶段必要性 | 必要 |
| 替换成本 | 低 |
| 自动化测试要求 | store 初始化、管理端与装置断开连接后保留当前页面进程内状态、重连刷新数据、退出清空会话等逻辑必须提供单元测试；不得通过 localStorage、IndexedDB 或文件系统保存业务状态。 |

## 2.9 管理端数据持久化策略

**选择：不做本地业务持久化**

| 维度 | 说明 |
|---|---|
| 为什么选 | PRD 明确要求管理端不持久化任何业务数据、账号数据、日志、最近 IP、当前页面进程内状态或查询结果。账号、角色、锁定状态、会话、三类日志、授权、策略均存储在 USB 装置侧。 |
| 允许的状态 | Pinia / Electron 主进程内存中保存当前连接状态、当前会话 token、当前角色、当前页面状态和当前请求结果；进程退出即丢失。 |
| 不选 SQLite / localStorage / IndexedDB / 文件日志 | 任意本地落盘都会与“管理端零业务持久化”原则冲突，并造成 USB 装置携带到不同管理主机时的数据不一致风险。 |
| 当前阶段必要性 | 必要 |
| 替换成本 | 低（不引入本地 DB 依赖） |
| 自动化测试要求 | 管理端自动化测试需增加本地持久化禁用检查，覆盖退出重启后账号、token、最近 IP、当前页面进程内状态、日志均不保留。 |

## 2.10 管理端与装置端通信协议

**选择：自定义私有协议（TCP + TLS + 二进制消息帧）**

| 维度 | 说明 |
|---|---|
| 为什么选 | 安全管控产品，通信内容包含策略配置、授权文件、升级包等敏感数据，使用私有协议可避免通用协议的扫描和攻击面。TLS 保证传输加密，自定义消息帧保证边界清晰。本版本按 PRD 采用命令/响应模式，不实现扫描进度推送。 |
| 协议结构 | `[4字节魔数][4字节消息类型][4字节序列号][4字节载荷长度][N字节载荷（Protobuf序列化）]` |
| 载荷序列化 | Protocol Buffers（Protobuf）。强类型、紧凑，装置端使用 `prost` crate（Rust），管理端使用 `protobufjs`（Node.js），版本兼容性好。 |
| 不选 HTTP REST | 标准协议，指纹明显，容易被网络扫描探测。且 HTTP 的请求/响应模式对装置侧主动推送支持不原生。 |
| 不选 gRPC | 本质是 HTTP/2 上的 Protobuf，HTTP 层面的限制同上，且 Web 端集成需要额外工具。 |
| TLS 证书方案 | 构建时生成自签名证书，随装置端安装包发布（证书 + 私钥内置在装置端程序中）；管理端安装包内置对应证书的 SHA256 指纹（硬编码）。连接时校验指纹一致性，不一致则拒绝并提示"版本不兼容，请升级管理端"。同一版本的装置和管理端使用同一张证书，升级后证书随安装包同步更新。账号鉴权、会话 token 和角色权限由装置端 API 执行。 |
| 详细设计要求 | 私有协议的消息类型、鉴权握手流程、重连机制、超时处理和错误码需在接口详细设计中完整定义，并同步生成协议测试用例。 |
| 当前阶段必要性 | 必要 |
| 替换成本 | 高（需要同步修改两端） |
| 自动化测试要求 | 必须提供协议 mock server、协议测试客户端和错误码测试集；覆盖正常响应、权限不足、未授权、连接断开、TLS 指纹不匹配、消息体格式错误和超时场景。 |

## 2.11 授权方案

**选择：基于机器码 + 离线授权文件**

| 维度 | 说明 |
|---|---|
| 为什么选 | 装置运行在内网隔离环境，无法在线校验授权。机器码由装置侧根据硬件信息（CPU ID/MAC 等）生成，用于唯一标识装置，授权文件由独立授权服务制作，本系统负责机器码生成、授权文件导入和装置侧授权校验接口封装。 |
| 本版本处理 | 架构上封装 `LicenseValidator` 授权校验接口。本版本使用简单模拟实现支撑页面流程、接口流程和权限流程联调；后续接入独立授权服务生成的正式授权文件时，不影响登录、授权页、系统管理页和协议主流程。 |
| 详细设计要求 | 机器码生成算法、授权文件字段结构、授权服务对接方式和正式校验规则由授权服务详细设计约束；本系统按文件导入和校验接口集成。当前 PRD 仅定义授权文件格式为 `.txt`。 |
| 当前阶段必要性 | 必要 |
| 自动化测试要求 | 当前通过模拟授权文件覆盖成功、失败、过期和格式错误分支；正式授权服务确定后补充授权文件生成、导入和校验自动化用例。 |

## 2.12 策略文件国密加密

**选择：SM4 对称加密 + SM3 摘要/完整性校验 + SM2 签名/验签**

| 维度 | 说明 |
|---|---|
| 为什么选 | 策略文件包含白名单和访问控制策略，需要满足国产密码算法要求，并同时具备保密、防篡改和防伪造能力。SM4 用于策略内容加密，SM3 用于摘要和完整性校验，SM2 用于策略文件签名/验签。 |
| 文件结构 | 文件头魔数 + 文件版本 + 算法标识 + 随机 IV/Nonce + SM4 密文 + SM3 摘要 + SM2 签名。版本兼容性通过文件头版本字段管理。 |
| 密钥方案 | SM4 对称密钥由装置侧安全配置持有或由构建/部署流程注入；SM2 公钥内置于装置端用于验签，SM2 私钥只保留在策略文件签发/导出侧。密钥不得出现在日志、导出内容明文或前端代码中。 |
| 导入校验顺序 | 先校验文件头和算法标识，再校验 SM2 签名和 SM3 摘要，最后使用 SM4 解密并反序列化策略内容。任一步骤失败均拒绝导入，原策略保持不变。 |
| 当前阶段必要性 | 必要 |
| 自动化测试要求 | 覆盖正确导入、签名错误、摘要错误、密文损坏、算法标识不支持、版本不兼容、解密失败、反序列化失败和原策略保持不变等用例。 |

## 2.13 部署方案

**装置端**

| 项目 | 方案 |
|---|---|
| 交付形态 | 装置端发布为离线安装包，采用 `.deb` 包或自解压 `.run` 安装包。安装包一次性完成服务程序、systemd unit、配置模板、数据库初始化脚本、USB Gadget 初始化脚本、NBD 依赖检查、ClamAV 运行环境部署/校验和版本信息写入。现场不得要求按步骤手工搭建运行环境。 |
| 安装包内容 | Rust aarch64 服务二进制、systemd service 文件、默认配置、SQLite schema/migration、协议证书、策略文件公钥、安装/升级/回滚/卸载脚本、健康检查脚本。 |
| 系统依赖 | Ubuntu 22.04、kernel 4.19、configfs、f_mass_storage、nbd kernel module、FAT32 / exFAT / NTFS / ext2 / ext3 / ext4 文件系统挂载能力、clamd 运行环境。基础系统镜像需预置或由安装包进行依赖校验；缺失关键依赖时安装失败并给出明确原因。 |
| 安装目录 | 程序文件位于 `/opt/usb-control/`；可变数据位于 `/var/lib/usb-control/`；运行时文件位于 `/run/usb-control/`；systemd unit 位于 `/etc/systemd/system/`。 |
| 进程管理 | systemd service 管理，开机自启，崩溃自动重启，集成 watchdog。安装完成后执行服务启动和健康检查，健康检查失败时回滚到安装前状态或保持旧版本运行。 |
| 安装规则 | 首次安装执行依赖检查、目录创建、配置初始化、数据库 schema 初始化、systemd unit 注册、服务启动和健康检查。安装失败时必须输出明确失败原因，并回滚已写入的程序文件和 systemd 配置。 |
| 程序升级入口 | USB 装置程序升级包由管理端“系统管理-系统升级”页面上传。装置端服务接收完整升级包后，先完成包格式、版本号、完整性和签名校验，校验通过后再交由本机 updater 执行替换。 |
| 程序升级规则 | USB 装置程序升级采用安装包覆盖升级。升级前仅保留当前服务二进制、配置文件、systemd unit 和安装元数据的旧版本副本，不全量备份账号、策略、授权、日志等业务数据。升级版本号必须大于当前版本，并遵循 PRD 中版本递增和逢 4 跳过规则。 |
| 升级执行方式 | 主服务完成升级包接收和校验后，启动独立 updater 进程或 systemd oneshot unit 执行程序替换。替换期间主服务可短暂不可用，管理端按与装置断开连接处理；升级完成后 updater 重启主服务并执行健康检查。 |
| 数据库处理 | 程序升级默认不执行数据库数据迁移。仅当升级包声明包含 schema 变更时，才执行随包提供的 migration 脚本；migration 必须使用 SQLite 事务、schema_version 记录和迁移日志。常规升级不得通过复制整库作为回滚手段。涉及大规模数据重写、不可逆迁移或需要额外临时空间的升级包，执行前必须完成空间预检查；空间不足或风险不可控时拒绝升级并保持旧版本运行。 |
| 回滚规则 | 程序替换、配置写入、服务启动或健康检查失败时，updater 自动恢复旧服务二进制、旧配置、旧 systemd unit 和旧安装元数据，确保装置回到升级前可运行状态。已提交的数据库 migration 不依赖全量数据备份回滚；如升级包包含不可逆 schema 变更，必须在详细设计中明确兼容策略、失败边界和人工恢复方案。 |
| 升级审计 | 安装、升级、回滚、卸载等维护动作写入装置侧操作日志；升级失败需记录失败阶段、错误码、旧版本、新版本和回滚结果。 |
| 卸载策略 | 普通卸载停止服务、移除程序文件、systemd unit、运行时文件和安装元数据；默认保留 `/var/lib/usb-control/` 中的账号、策略、授权、日志等业务数据，避免误删装置侧权威数据。 |
| 数据清除 | 彻底清除业务数据必须作为独立维护动作，不与普通卸载合并；执行前需二次确认，并明确清除账号、策略、授权、三类日志和系统配置后的恢复方式。 |
| 资源预算 | 4GB RAM / 32GB 存储条件下，Ubuntu 22.04 最小化系统、ClamAV、Rust 服务、SQLite 数据库、三类业务日志和 Rust 服务运行日志需满足长期运行要求。装置整体存储使用率超过 80% 后，三类业务日志按 PRD 同类别滚动覆盖，Rust 服务运行日志按自身本地文件滚动策略清理或覆盖最旧运行日志文件。 |

**管理端**

| 项目 | 方案 |

## 2.14 日志与监控方案

**装置端**

- USB 审计日志、恶意代码检测日志、操作日志：结构化存储在装置侧 SQLite，管理端连接后拉取展示、查询、导出和清理
- Rust 服务运行日志：以本地文件方式保存，用于装置端程序运行、调试和异常排查，可包含 `trace`、`debug`、`info`、`warn`、`error` 等级；不属于三类业务日志，不在管理端日志管理页面展示、查询、导出或清理
- 装置整体存储使用率超过 80% 时触发日志空间控制：三类业务日志按当前写入类别覆盖该类别最旧记录，不跨类覆盖；Rust 服务运行日志按自身本地文件滚动策略清理或覆盖最旧运行日志文件
- 装置端数据库文本字段、策略内容、日志内容和协议字符串统一按 UTF-8 存储和传输

**管理端**

- 不落盘保存运行日志、异常日志、操作日志或业务日志
- 管理端运行异常仅在界面提示；未连接装置时发生的异常不持久化
- 日志导出 `.zip` 内如包含中文文件名或中文文本内容，文件名编码需使用 UTF-8；日志内容文件统一使用 UTF-8，避免在 Windows 简体/繁体环境或第三方解压工具中出现乱码

**本版本不引入** Prometheus、ELK 等集中式监控平台。装置端日志由 SQLite 三类业务日志和本地文件 Rust 服务运行日志组成，管理端遵循不落盘原则。
