# 07 权限模型

---

## 1. 三权分立设计

系统采用三角色权限分离设计，每个角色只能访问本职职责范围内的功能，角色之间无功能交叉。

| 角色 | 英文标识 | 职责范围 | 人员定位 |
|---|---|---|---|
| 系统管理员 | admin | 系统维护：升级、授权、设备描述、用户管理 | IT 管理人员 |
| 操作员 | operator | 管控配置：U 盘白名单、文件访问控制、策略导入导出 | 安全运维人员 |
| 审计员 | auditor | 日志审计：查询/导出/清理三类日志 | 合规审计人员 |

**三权分立的核心约束：**
1. 操作员不可以查看日志（由审计员负责）
2. 审计员不可以修改策略（由操作员负责）
3. 系统管理员不可以修改策略或查看日志（由操作员和审计员分别负责）
4. 所有角色均可修改自己的密码

## 4. 内置账号规则

| 账号名 | 角色 | 初始密码 | 是否可删除 | 说明 |
|---|---|---|---|---|
| admin | 系统管理员 | `admin@123` | 不可删除 | 内置系统管理员 |
| operator | 操作员 | `operator@123` | 不可删除 | 内置操作员 |
| audit | 审计员 | `audit@123` | 不可删除 | 内置审计员 |

**内置账号约束：**
- 在用户管理页面，内置账号无"删除"操作按钮
- 内置账号可以修改密码（通过修改密码功能）
- 内置账号可以被锁定（5次登录失败），被锁定时系统管理员可重置密码

---

## 5. 账号锁定规则

| 规则 | 值 | 说明 |
|---|---|---|
| 锁定触发条件 | 连续 5 次登录失败 | 包含用户名/密码错误 |
| 锁定时长 | 5 分钟 | 超时后自动解锁 |
| 锁定状态持久化 | 是 | 存储在装置侧，服务重启不清除锁定状态 |
| 解锁方式 | 锁定 5 分钟后自动解锁 | 不提供提前解锁入口 |
| 锁定期间行为 | 任何凭据均提示"用户已被锁定，请过5分钟后重试" | |
| 失败计数清零 | 登录成功、锁定超时自动解除或系统管理员重置密码后清零 | |

## 6. 密码规则

密码复杂度要求：

- 最小长度：8 位
- 必须包含英文字母、数字、特殊字符中的至少两类
- 内置账号初始密码写死（`admin@123` / `operator@123` / `audit@123`），符合上述规则
- 新建账号、重置密码和修改密码均须满足此规则，不符合时禁止保存并提示
- 新建账号和重置密码表单均需要确认密码，两次输入不一致时禁止提交

## 7. 会话管理规则

| 规则 | 值 | 说明 |
|---|---|---|
| 会话超时时长 | 5 分钟无操作 | PRD 明确 |
| 超时行为 | 装置侧会话失效；管理端跳转登录页并断开 TCP 连接 | |
| 会话存储 | 装置侧维护会话 token；管理端仅在 Pinia/主进程内存保存 token、当前用户名和角色，不持久化到磁盘 | 关闭程序后需重新登录 |
| 会话范围 | 装置端为权威会话；管理端只保存当前进程内会话副本 | |
| 单一连接 | 每个装置同时只允许一个管理端 TCP 连接 | 装置侧 TCP 服务保证，第二个连接尝试时拒绝 |

## 8. 权限校验层次

系统采用“管理端 UX 校验 + 装置端强制校验”两层权限：

```
用户操作 → 前端路由守卫（C01）+ 权限展示状态（C02）
              ↓ 通过（角色有权访问，UX 层）
          前端 API 调用
              ↓
          C03 通过已建立的 TLS 连接发送命令 + session token 到装置端
              ↓
          装置端 S10/S08 校验 token、授权状态、角色权限
              ↓ 通过
          装置端业务模块执行并写入操作日志
```

**前端权限（C01/C02）：** 控制菜单可见、路由访问、按钮显示。防止正常用户误操作，属于 UX 层面的保护。角色信息来自装置端登录响应，仅保存在管理端进程内存。

**装置端权限（S10/S08）：** 对所有登录后接口校验 session token、授权状态和角色权限；权限不足返回统一错误并写入装置侧操作日志。TLS 证书固定用于保护连接来源，不能替代用户级鉴权。

## 9. 已删除用户名复用规则

| 场景 | 处理 |
|---|---|
| 删除用户名后新建相同用户名 | 拒绝，提示"该用户名不可使用" |
| 原因 | 操作日志记录的历史操作人信息须保持可追溯性 |
| 实现 | 用户表中被删除的用户名保留（状态标记为 DELETED），新建时检查 DELETED 状态 |
