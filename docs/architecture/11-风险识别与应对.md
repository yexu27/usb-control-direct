## 说明

## R01 NBD 虚拟 FAT32 扇区生成的实现复杂度与性能

| 维度 | 内容 |
|---|---|
| 风险描述 | 文件访问控制链路为：真实 U 盘 → kernel mount 到 `/mnt/usb_raw` → Rust NBD 虚拟 FAT32 扇区服务器 → f_mass_storage。NBD FAT32 虚拟扇区生成、写入映射和 I/O error 反馈是整套方案中实现复杂度最高的部分，存在工期风险 |
| 影响 | 高（影响核心功能可用性与研发工期）|
| 概率 | 中（NBD FAT32 生成器需从头实现，且需验证 Windows/Linux 对 READ 返回 I/O error 的实际表现）|
| 应对策略 | 1. 技术预研阶段在真实 RK3568 上测试 kernel mount + NBD 链路端到端读写性能（大文件拷贝场景）；2. 独立验证 NBD READ 返回 I/O error 后 Windows/Linux 的弹窗和重试行为；3. NBD FAT32 生成器独立预研，评估 `fatfs` crate 可利用程度；4. 实现分阶段：先实现只读和阻断读取场景，再实现写场景 |
| 处理阶段 | 技术预研阶段，研发启动前完成可行性验证 |

## R02 ClamAV 在 RK3568 上的扫描性能

| 维度 | 内容 |
|---|---|
| 风险描述 | PRD 明确要求 ClamAV 全盘扫描，但 RK3568 ARM 性能有限，大容量 U 盘（128GB+）全盘扫描可能耗时数十分钟，严重影响使用体验 |
| 影响 | 高（扫描未完成前 U 盘不能使用）|
| 概率 | 高（ClamAV 全盘扫描 CPU 密集）|
| 应对策略 | 1. 在 RK3568 上测量真实扫描速度（EICAR + 实际文件）；2. 评估大容量 U 盘扫描耗时对使用体验的影响；3. 本版本仍按 PRD 执行全盘扫描，不增加主动进度通知；4. 扫描期间仅按 PRD 提示扫描状态，完成后再进入映射流程 |
| 处理阶段 | 技术预研阶段 |
